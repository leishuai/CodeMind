# iOS External XCUITest Runner Summary

Use an external XCUITest runner when the target app has no usable UI test target/scheme.

## Operating rules (read first)

- **Use case:** choose the external UI runner when the project has no usable UI
  test target and editing the original `.xcodeproj` would be risky. The runner
  drives an already-installed target app by bundle id.
- **Position in ladder:** try existing project-native UI test target and
  decoupled native XCUITest first; use external UI runner as the no-edit
  real-device path before temporary project edits, simulator downgrade, or
  human-assisted operation.
- **Real-device status:** external UI runner can work on an ordinary connected
  iPhone. Do not pre-classify it as device-unavailable just because the project
  lacks a native UI test target.
- **Simulator status:** external UI runner also supports iOS Simulator execution,
  but simulator evidence is not real-device proof unless the testcase allows it
  or a runtime downgrade is approved.
- **Required routing data:** keep CoreDevice ID and traditional UDID separate;
  use CoreDevice ID for `devicectl install/launch`, and traditional UDID for
  `xcodebuild -destination id=`.
- **Target app channel:** the target bundle id must reach the on-device runner
  process through `TEST_RUNNER_AUTOMIND_TARGET_BUNDLE_ID` (or
  `--target-bundle-id` / `iosApp.targetBundleId`). A plain shell env var is not
  inherited by the runner process.
- **Avoid path:** do not retry `pymobiledevice3 dvt xcuitest` as a host for an
  arbitrary IDE-dependent custom runner. Use a project/native Xcode channel or a
  WDA/go-ios/Appium-style channel that actually services UI commands.
- **Root-install blocker:** `Root install style is not supported` on a retail
  physical device is a runner-delivery issue, not proof that the device or the
  external UI runner is unusable. Replan to another real-device runner strategy;
  do not downgrade a real-device testcase to dry-run.

## Why

Some real projects have app targets but no generated UI test target. Instead of editing complex project files, a small external runner can drive an already-installed app by bundle id:

```swift
let app = XCUIApplication(bundleIdentifier: "com.example.app")
app.launch()
```

## Verified physical-device success path (re-confirmed 2026-06-12)

External UI runner DOES work on an ordinary connected iPhone. Do not pre-classify physical-device UI automation as "device unavailable"; split iOS real-device verification into stages and verify each: **device discovery -> signing/provisioning -> host-app build/install -> XCUITest runner launch -> UI automation execution**.

1. Device discovery (read-only). `xcrun devicectl list devices` shows the iPhone `connected` with a **CoreDevice ID** (a UUID such as `<CoreDeviceID>`); `idevice_id -l` shows the **traditional UDID** (such as `<UDID>`). These are two different ID namespaces:
   - `devicectl device install app --device <CoreDeviceID>` wants the CoreDevice ID.
   - `xcodebuild -destination id=<UDID>` is most reliable with the traditional UDID.
2. Build the host app: `xcodebuild build -project demos/ios-external-ui-runner/AutoMindExternalUIRunner.xcodeproj -scheme AutoMindExternalUIRunner -destination id=<UDID> DEVELOPMENT_TEAM=<TEAM_ID> CODE_SIGN_STYLE=Automatic -allowProvisioningUpdates`. Automatic signing + `-allowProvisioningUpdates` lets Xcode create/refresh the profile for both the host app and the `.xctrunner`; the log shows both embedding `iOS Team Provisioning Profile: ai.openclaw.automind.external.demo`.
3. (Optional) Install standalone via `xcrun devicectl device install app --device <CoreDeviceID> <AutoMindExternalUIRunner.app>` to confirm the bundle id lands on the device. `xcodebuild test` will also build+install+launch on its own.
4. Run `xcodebuild test` with the same `-destination id=<UDID> DEVELOPMENT_TEAM=... CODE_SIGN_STYLE=Automatic -allowProvisioningUpdates` args. Success markers in order: profiles embedded -> `Running tests...` -> `testExternalTargetLaunches` opens `ai.openclaw.automind.external.demo` via `XCUIApplication(bundleIdentifier:).launch()` -> waits for app idle -> `Executed 1 test, with 0 failures` -> `** TEST SUCCEEDED **`.
5. CodeAutonomy's own runner reproduces it with the same params: `scripts/ios_xcuitest_runner.py <task> --project-path demos/ios-external-ui-runner/AutoMindExternalUIRunner.xcodeproj --scheme AutoMindExternalUIRunner --device-id <UDID> --team <TEAM_ID>` -> `evaluation.json` `result=pass`, `nextAction=finish`.

Core takeaway: external UI runner is feasible on an ordinary connected real device, given (a) the correct device-ID mapping (CoreDevice ID vs traditional UDID), (b) an explicitly passed Team, (c) Automatic provisioning enabled, and (d) the user tapping through any on-device prompt promptly.

## Detailed rules and avoid paths

- New runner bundle ids may fail provisioning if Xcode account credentials cannot create profiles.
- Reusing an existing known-good test profile avoids that blocker.
- CoreDevice ID (from `devicectl list devices`) and traditional UDID (from `idevice_id -l`) are different namespaces: use CoreDevice ID for `devicectl install/launch`, traditional UDID for `xcodebuild -destination id=`.
- An on-device prompt for the iPhone passcode / "trust" during the run is NOT a failure — it is part of the success path. The user just needs to enter the passcode and keep the device unlocked/trusted, then the XCUITest runner launches. (When the profile is already trusted, the run can proceed with no prompt at all.)
- Keep XCUITest focused on actions it can reliably perform; use screenshot/OCR/app-smoke evidence for post-action readiness.
- Avoid invalid predicate keys such as `hittable` inside `NSPredicate`; check hittability in code instead.
- Before executing a product journey, encode the task intent in `probe-flow.ios.json` (`testIntent.goal/sources/steps/acceptanceCriteria`) and run `./automind.sh ios-probe-flow <task> --dry-run` so the path is reviewable instead of invented by the runner.
- On Xcode 26 + iOS 18.x retail devices, `xcodebuild test-without-building` does NOT hit "Root install style not supported (hasInternalOSBuild:false)"; that line, if seen, is a runner-delivery issue (`external_runner_root_install_unsupported`, retry_generator), not a device-capability/root blocker. It also does not invalidate the external UI runner, which supports iOS Simulator execution. Do not ask the user to fix the device.
- `pymobiledevice3` (9.17) `dvt xcuitest` does NOT implement `XCTestManager_IDEInterface`, so it cannot host an *arbitrary IDE-dependent* runner — that runner starts then disconnects (`channel canceled for XCTestManager_IDEInterface` / `Exiting due to IDE disconnection`, `external_runner_capability_blocked`, replan). This is scoped to hosting a runner that expects the Xcode IDE on the other end; it does NOT mean pymobiledevice3 cannot drive UI at all. On iOS 17+ pymobiledevice3's `developer` tunnel + `XCUITestService` can launch/manage **WebDriverAgent**, and UI actions (tap/input) then go through WDA's own WebDriver/HTTP endpoint (no Xcode IDE channel needed) — this is the standard Appium/go-ios way to drive iOS 17+ real devices. So: to drive an external runner you need either a complete IDE-side channel (project UI test target via Xcode), or a WDA-based runner whose commands ride WDA's server. Do not retry the exact `dvt xcuitest`-hosts-our-custom-runner path.
- External-runner install mechanism: prefer `devicectl device install app` + `devicectl device process launch` for the app under test, and run the UI test bundle through a runner that carries its own IDE-side interface; do not assume `pymobiledevice3 dvt xcuitest` will host the test session.
- Once the session has actually started (`Running tests...`), a later DTX connection drop (`DTXConnection` / `Lost connection to the test process` / `The connection was lost`) or process `exit code 74` is a test-runtime disconnect, NOT a signing/provisioning failure: the runner was signed and launched far enough to begin the session. The classifier routes this to `test_failure` / `retry_generator` (`ios.xcuitest.session.dtx_disconnect_or_code74`). Investigate the test process crash/hang, device stability, or the test harness, and retry — do not re-sign or ask the user for signing material. Code-sign failures only count as signing when they happen *before* the `Running tests...` marker.
- A connection drop *before* `Running tests...` (`Lost connection to the device`, `Could not connect to lockdown`, `Failed to start the test runner`) is a **device/host link instability**, not signing, code, or device-absence. The classifier self-retries it first (`ios.device.link_lost_before_session`, `retry_generator`); if it keeps dropping (context `deviceLinkRetryExhausted`) it asks the user to physically recover the link — **keep the iPhone unlocked and lit, replug the USB cable (try another port/cable), accept any trust prompt, toggle Settings -> Developer -> Enable UI Automation off and on, restart Xcode/CoreDevice if needed, or reboot the iPhone and re-trust the Mac** (`ios.device.link_lost_repeated`, `ask_user`, category `system_or_external_dependency`). Do not re-sign or treat it as "device unavailable".
- Portability: the device IDs (`<CoreDeviceID>`, `<UDID>`) and `DEVELOPMENT_TEAM=<TEAM_ID>` in this doc are placeholders for **the local machine's** values, kept generic so no personal device/team identifiers are stored here. On any Mac/device, discover them fresh (`xcrun devicectl list devices`, `idevice_id -l`, `automind ios-signing-preflight <task> --discover` for the Team) instead of hardcoding — the runner already auto-fills the Team from discovered signing material when it is empty. The reusable part is the *procedure and the ID-namespace mapping*, not the literal IDs.
- Target app bundle id (the app the runner should drive, e.g. a real product like `com.example.app`) must reach the **on-device** XCUITest runner process. A plain shell env var (`AUTOMIND_TARGET_BUNDLE_ID=...`) is NOT inherited by that process — the runner keeps opening its default demo app and the log shows the wrong app launching. The reliable channel is `xcodebuild test`'s `TEST_RUNNER_<VAR>` forwarding: pass `TEST_RUNNER_AUTOMIND_TARGET_BUNDLE_ID=<bundle>` and it arrives inside the runner as `AUTOMIND_TARGET_BUNDLE_ID`, which the runner reads for `XCUIApplication(bundleIdentifier:)`. `scripts/ios_xcuitest_runner.py` does this automatically via `--target-bundle-id <bundle>` (or `iosApp.targetBundleId` in task state); the demo runner falls back to its own bundle id only when the var is absent. Confirm success by `Open <bundle>` / `Launch <bundle>` lines for the *intended* app, not the demo.

## Example result shape

A successful external runner should launch the configured target bundle id, handle only explicitly authorized blockers, and verify the target remains running with XCUITest evidence.
